📖 詳細解説

== と equals の違い

📌 結論文字列の内容を比較するには必ず equals() を使う。== は参照（アドレス）の比較。

基本説明

==演算子はプリミティブ型では値比較、参照型ではオブジェクトのアドレスを比較します。
equals()はオブジェクトの「内容」を比較します（Stringはオーバーライド済み）。

Java: == vs equals
String s1 = "Hello";
String s2 = "Hello";          // 文字列プールから同じオブジェクト
String s3 = new String("Hello"); // 新しいオブジェクト

System.out.println(s1 == s2);       // true（同じアドレス）
System.out.println(s1 == s3);       // false（別アドレス）
System.out.println(s1.equals(s3));  // true（内容が同じ）

// NullPointerException を避けたい場合
String input = null;
System.out.println("Hello".equals(input));     // false（NPEなし）
System.out.println(java.util.Objects.equals(s1, input)); // false
⚠️ 試験で狙われるポイント new String() で生成したオブジェクトと == で比較するとfalseになる。 nullのオブジェクトにequals()を呼ぶとNullPointerExceptionが発生するため、 リテラルを左辺に置く（"Hello".equals(s)）か Objects.equals()を使う。

実務での使い方

APIレスポンスや入力値の文字列比較には常に equals() を使います。
enumの比較は == でOKです（enumは同じインスタンスが保証されるため）。

String と StringBuilder の違い

📌 結論String は不変（イミュータブル）。StringBuilder は可変（ミュータブル）。ループ内の連結はStringBuilderを使う。

基本説明

Stringは一度作ると内容を変更できません（+=のたびに新オブジェクト生成）。
StringBuilderは同じオブジェクトの内部バッファに追記するため効率的です。

Java: StringBuilderの主要メソッド
StringBuilder sb = new StringBuilder("Hello");

sb.append(" World");    // Hello World
sb.insert(5, ",");       // Hello, World
sb.delete(5, 6);        // Hello World
sb.reverse();             // dlroW olleH
sb.replace(0, 5, "Hi"); // Hi World（0〜4を置換）

System.out.println(sb.toString()); // String変換
System.out.println(sb.length());   // 文字数
System.out.println(sb.charAt(0)); // 'H'
⚠️ 試験で狙われるポイント StringBufferはStringBuilderのスレッドセーフ版（synchronized）で低速。 シングルスレッドではStringBuilder、マルチスレッドではStringBufferを使う。 Silver試験ではStringBuilderのメソッドの動作が頻出。

static の意味

📌 結論static = クラスに属する。インスタンスなしで使える。全インスタンスで共有される。

基本説明

通常のフィールド・メソッドはインスタンス（オブジェクト）に属します。
staticをつけるとクラスに属し、インスタンスを作らずに使えます。

Java: staticの使い方
class MathUtil {
    static final double PI = 3.14159; // 定数（変更不可）

    static double circle(double r) {   // インスタンス不要
        return PI * r * r;
    }
}

// インスタンスを作らずに呼べる
System.out.println(MathUtil.PI);
System.out.println(MathUtil.circle(5.0));

// staticメソッドからインスタンス変数は使えない！
class Broken {
    int x = 10;
    static void method() {
        System.out.println(x); // コンパイルエラー！
    }
}
⚠️ 試験で狙われるポイント staticメソッドからはinstanceメンバにアクセス不可。 staticフィールドはすべてのインスタンスで共有。 staticメソッドのオーバーライドは不可（hidingになる）。

final の意味

📌 結論final = 変更禁止。変数なら再代入不可、メソッドならオーバーライド不可、クラスなら継承不可。
Java: finalの3つの使い方
// 1. final変数（再代入不可）
final int MAX = 100;
MAX = 200; // コンパイルエラー！

// ただし参照型の場合、中身の変更は可能
final List<String> list = new ArrayList<>();
list.add("OK"); // これはOK（listの中身を変えているだけ）
list = new ArrayList<>(); // コンパイルエラー！（listへの再代入がNG）

// 2. finalメソッド（オーバーライド不可）
class Parent {
    final void doSomething() { /* ... */ }
}
class Child extends Parent {
    void doSomething() { } // コンパイルエラー！
}

// 3. finalクラス（継承不可）
final class Immutable { }
class Sub extends Immutable { } // コンパイルエラー！
// 例: java.lang.String は finalクラス

継承とオーバーライド

📌 結論継承（extends）は親クラスの機能を引き継ぐ。オーバーライドは子クラスで親のメソッドを上書きする。実行時に実体の型で呼ぶメソッドが決まる。
Java: 継承とオーバーライドの完全な例
class Shape {
    String color;
    Shape(String color) { this.color = color; }
    double area() { return 0; }
    String describe() { return color + " shape, area=" + area(); }
}

class Circle extends Shape {
    double radius;
    Circle(String color, double r) {
        super(color);  // 親コンストラクタ呼び出し（第1行目）
        this.radius = r;
    }
    @Override
    public double area() { return 3.14 * radius * radius; }
}

Shape s = new Circle("red", 5);
System.out.println(s.describe());
// "red shape, area=78.5"
// describe()内でarea()を呼ぶと、オーバーライドされたCircle.area()が実行される

instanceof 演算子

System.out.println(s instanceof Shape);  // true
System.out.println(s instanceof Circle); // true

// ダウンキャスト（ClassCastExceptionに注意）
Circle c = (Circle) s;  // OK（実体がCircleなので）
c.radius;               // Circleのフィールドにアクセス可能

抽象クラスとインターフェースの違い

📌 結論「is-a（〜である）」関係 → 抽象クラス。「can-do（〜できる）」関係 → インターフェース。多重実装が必要なら → インターフェース。
Java: 比較例
// 抽象クラス：共通の状態と処理を持つ基底クラス
abstract class Employee {
    String name;
    double baseSalary;
    Employee(String name, double base) {
        this.name = name; this.baseSalary = base;
    }
    abstract double calcSalary();     // 各子クラスで実装必須
    void printInfo() {                 // 共通実装
        System.out.println(name + ": " + calcSalary());
    }
}

// インターフェース：機能の契約
interface Printable {
    void print();     // 実装クラスが必ず提供
    default void preview() { System.out.println("Preview..."); }
}
interface Exportable {
    void export(String path);
}

// 多重実装が可能
class Report extends Employee implements Printable, Exportable {
    Report(String name) { super(name, 300000); }
    public double calcSalary() { return baseSalary * 1.2; }
    public void print() { System.out.println("Report: " + name); }
    public void export(String path) { System.out.println("Export to: " + path); }
}

例外処理

📌 結論チェック例外はthrowsかcatchが必須。非チェック例外（RuntimeException）は任意。finallyは必ず実行される。

例外の階層

Throwable
├── Error  ← 通常キャッチしない（OutOfMemoryError等）
└── Exception
    ├── RuntimeException（非チェック例外）← throws不要
    │   ├── NullPointerException（null参照）
    │   ├── ArrayIndexOutOfBoundsException（配列範囲外）
    │   ├── ClassCastException（不正キャスト）
    │   ├── NumberFormatException（数値変換失敗）
    │   └── ArithmeticException（0除算等）
    └── その他Exception（チェック例外）← throws or catch 必須
        ├── IOException
        ├── SQLException
        └── FileNotFoundException
Java: try-catch-finallyの完全な例
try {
    String s = null;
    s.length();  // NullPointerException発生！
} catch (NullPointerException e) {
    System.out.println("NPE: " + e.getMessage());
} catch (Exception e) {
    System.out.println("その他の例外");  // より広い例外は後に書く
} finally {
    System.out.println("必ず実行");  // 例外有無にかかわらず実行
}

// カスタム例外の作り方
class InsufficientFundsException extends Exception {
    double amount;
    InsufficientFundsException(double amount) {
        super("残高不足: " + amount);
        this.amount = amount;
    }
}

ArrayList

📌 結論ArrayListは動的サイズの配列。サイズ変更可能で、インデックスアクセスがO(1)と高速。Listインターフェースを実装している。
Java: ArrayListの主要操作
List<String> list = new ArrayList<>();

// 追加
list.add("Apple");           // 末尾に追加
list.add(0, "Avocado");     // 指定位置に挿入

// 取得・確認
String s = list.get(0);      // "Avocado"
int size = list.size();       // 要素数
boolean has = list.contains("Apple"); // true
int idx = list.indexOf("Apple"); // 最初のindexを返す

// 変更・削除
list.set(0, "Mango");        // 0番目を置き換え
list.remove(0);              // index指定で削除
list.remove("Apple");        // 値指定で削除（最初の1件）

// ループ
for (String item : list) {   // 拡張for文
    System.out.println(item);
}

// ソート
Collections.sort(list);       // 昇順
list.sort((a, b) -> b.compareTo(a)); // 降順（ラムダ）
⚠️ よくある間違い remove(int index) と remove(Object o) の区別に注意。 List<Integer> の場合、remove(0) はindex削除、remove(Integer.valueOf(0)) は値削除。

ラムダ式

📌 結論ラムダ式は「関数型インターフェース（抽象メソッドが1つのinterface）」の実装をシンプルに書く構文。Java8以降で使用可能。

基本構文

// 通常の匿名クラス
Runnable r1 = new Runnable() {
    public void run() { System.out.println("run!"); }
};

// ラムダ式（省略形）
Runnable r2 = () -> System.out.println("run!");

// 引数あり・処理が複数行
Comparator<String> comp = (a, b) -> {
    return a.length() - b.length();
};

// メソッド参照（さらに省略可能）
list.forEach(System.out::println); // 各要素をprintln

主な関数型インターフェース

// Predicate<T>: T → boolean
Predicate<Integer> isEven = n -> n % 2 == 0;
System.out.println(isEven.test(4)); // true

// Function<T,R>: T → R
Function<String, Integer> len = s -> s.length();
System.out.println(len.apply("Hello")); // 5

// Consumer<T>: T → void
Consumer<String> print = s -> System.out.println(s);
print.accept("Hello");

// Supplier<T>: () → T
Supplier<String> greet = () -> "Hello!";
System.out.println(greet.get()); // "Hello!"

// Stream APIとの組み合わせ
List<String> names = Arrays.asList("Alice", "Bob", "Carol");
names.stream()
    .filter(n -> n.length() > 3)   // Predicate
    .map(String::toUpperCase)         // Function（メソッド参照）
    .forEach(System.out::println);    // Consumer（メソッド参照）
✅ 試験対策まとめ ① ラムダ式は関数型インターフェース（@FunctionalInterface）にのみ使える
② Predicate/Function/Consumer/Supplierのシグネチャを覚える
③ Stream APIの中間操作（filter/map/sorted）と終端操作（collect/count/forEach）を区別する
④ メソッド参照は「クラス名::メソッド名」の形式
