💼 業務で役立つJava知識

Java Silver の知識が実際の開発現場でどう活きるかを解説します

📌 このページの目的 試験勉強で学ぶ知識が「なぜ実務で重要なのか」を理解することで、 単なる暗記を超えた深い理解が得られます。理解が深まると試験の正答率も上がります。
if文 for文 while文 ★★★ 最重要

① 制御構文はすべての業務ロジックの基礎

業務システムでは、条件に応じた処理分岐（if/switch）と繰り返し（for/while）が至る所で使われます。 受注管理、在庫チェック、ユーザー認証など、あらゆるビジネスロジックが制御構文で書かれています。

Java: 実務的な制御構文の例
// 受注ステータスによる処理分岐
public String getStatusLabel(String status) {
    return switch (status) {         // Java14以降のswitch式
        case "PENDING"   -> "受付中";
        case "CONFIRMED" -> "確認済み";
        case "SHIPPED"   -> "発送済み";
        case "CANCELLED" -> "キャンセル";
        default          -> "不明";
    };
}

// 一括処理でfor文を使う（バッチ処理の基本形）
for (Order order : orders) {
    if (order.isExpired()) {
        order.cancel();
        notifyCustomer(order);
    }
}
クラス設計 カプセル化 継承

② クラス設計は保守性・拡張性に直結する

適切なクラス設計は、将来の機能追加・バグ修正のしやすさを大きく左右します。 カプセル化（private + getter/setter）によって、内部の変更が外部に影響しないコードが書けます。

Java: カプセル化の実践例
class Product {
    private String name;
    private int price;   // 直接アクセス不可（外部から変更させない）
    private int stock;

    public String getName() { return name; }
    public int getPrice() { return price; }

    // セッターにバリデーションを入れられる
    public void setPrice(int price) {
        if (price < 0) throw new IllegalArgumentException("価格は0以上");
        this.price = price;
    }

    public boolean isInStock() { return stock > 0; }
}
実務のポイント フィールドはすべてprivateが基本。 IDEでgetter/setterは自動生成できます（Eclipse: Alt+Shift+S / IntelliJ: Alt+Insert）。 Lombokライブラリを使うと@Getterアノテーションで自動生成も可能。
例外処理 障害対応 ログ ★★★ 最重要

③ 例外処理は障害対応・品質の鍵

本番環境での障害調査には、適切な例外処理とログ出力が不可欠です。 例外を握りつぶす（空のcatch）と原因が分からなくなります。 finallyまたはtry-with-resourcesでリソースを確実に解放することも重要です。

Java: 実務的な例外処理
public List<User> getUsers(int departmentId) {
    // try-with-resourcesでDB接続を自動クローズ
    try (Connection conn = dataSource.getConnection();
         PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE dept_id=?")) {
        ps.setInt(1, departmentId);
        ResultSet rs = ps.executeQuery();
        List<User> users = new ArrayList<>();
        while (rs.next()) {
            users.add(mapToUser(rs));
        }
        return users;
    } catch (SQLException e) {
        // ログに記録（握りつぶしNG！）
        log.error("DBエラー: dept_id={}", departmentId, e);
        throw new ServiceException("ユーザー取得失敗", e);
    }
}
⚠️ 実務でやってはいけないこと
catch (Exception e) { } // 空のcatch → 原因が闇に消える
catch (Exception e) { e.printStackTrace(); } // 標準出力への出力（本番NGの場合が多い）
→ 必ずロガー（SLF4J + Logback等）を使ってログを記録する
ArrayList HashMap Stream 一覧処理

④ コレクションは一覧データ処理で毎日使う

Web APIやDBから取得したデータはList・MapなどのコレクションとしてJavaで扱います。 Stream APIを組み合わせることで、SQLライクな宣言的な処理が書けます。

Java: Stream APIによる実務的なデータ処理
// DBから取得したユーザーリストを加工する実務的な例
List<User> users = userRepository.findAll();

// 有効なユーザーだけ取り出し、名前順でソート
List<User> activeUsers = users.stream()
    .filter(User::isActive)
    .sorted(Comparator.comparing(User::getName))
    .collect(Collectors.toList());

// 部署ごとにグルーピング（SQLのGROUP BYに相当）
Map<String, List<User>> byDept = users.stream()
    .collect(Collectors.groupingBy(User::getDepartment));

// 平均年齢を計算（SQLのAVGに相当）
OptionalDouble avgAge = users.stream()
    .mapToInt(User::getAge)
    .average();
avgAge.ifPresent(a -> System.out.println("平均年齢: " + a));

Interface Spring Boot DI テスタビリティ

⑥ Interfaceは疎結合設計の要

Spring BootではService・Repositoryの設計にInterfaceが多用されます。 インターフェースで抽象化することで、テスト時にモックに差し替えやすくなります。

Java: Spring Bootでのインターフェース活用
// インターフェースで契約を定義
public interface UserService {
    User findById(long id);
    List<User> findAll();
    void save(User user);
}

// 本番実装
@Service
public class UserServiceImpl implements UserService {
    @Autowired
    private UserRepository repo;

    public User findById(long id) { return repo.findById(id).orElseThrow(); }
    // ...
}

// テスト用モック実装（DIで差し替え可能）
public class MockUserService implements UserService {
    public User findById(long id) { return new User("テスト"); }
    // ...
}
設計原則 「具体クラスではなくインターフェースに依存せよ」（依存性逆転の原則）。 Springの@Autowiredはインターフェース型で注入することが推奨されます。
static アンチパターン テスタビリティ

⑦ staticの乱用は避けるべき

staticは便利ですが、乱用するとテストが難しくなり、マルチスレッドでのバグを招きます。 ユーティリティメソッドや定数にのみ使うのが適切です。

Java: staticのアンチパターンと対策
// ❌ アンチパターン: staticで状態を持つ
class BadCache {
    static Map<String, User> cache = new HashMap<>();
    // マルチスレッドで競合・テストで状態が残る
}

// ✅ 良いパターン: インスタンス管理 + DIで注入
@Component
class UserCache {
    private final Map<String, User> cache = new ConcurrentHashMap<>();
    public User get(String key) { return cache.get(key); }
    public void put(String key, User user) { cache.put(key, user); }
}

// ✅ staticが適切な場面：状態を持たないユーティリティ
class StringUtils {
    static boolean isNullOrEmpty(String s) {
        return s == null || s.isEmpty();
    }
}
